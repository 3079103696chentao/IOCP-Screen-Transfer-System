# IOCP画面传输系统

这是一个基于 Windows C++ / MFC 的远程画面传输与控制项目。项目以 IOCP 异步网络模型为核心，打通了“客户端发起命令 -> 服务端异步收包 -> 自定义协议解析 -> 命令执行 -> 结果封包回传 -> 客户端界面展示”的完整链路。

> 说明：本项目仅用于本地实验、课程学习和面试展示，请只在自己拥有或明确授权的机器和网络环境中运行。

## 简历对应描述

可以对应简历中的如下项目描述：

> 基于 Windows C++ / MFC 实现 IOCP 画面传输系统，服务端使用 `AcceptEx`、`WSARecv`、`WSASend` 和完成端口处理异步网络事件；设计自定义二进制协议解决 TCP 粘包、半包和命令分发问题；客户端和服务端按 `task / screen / control` 拆分通信与执行链路，支持连接测试、远程目录浏览、文件下载/删除、屏幕画面采集传输、鼠标控制和锁机/解锁功能。

## 核心功能

- IOCP 异步服务端：基于 `CreateIoCompletionPort`、`AcceptEx`、`WSARecv`、`WSASend` 实现连接接入、异步接收和异步发送。
- 自定义协议：通过包头、长度、命令字、数据区和校验和封装应用层数据，处理 TCP 字节流中的粘包和半包。
- 远程画面传输：服务端使用 GDI / GDI+ 采集桌面画面并保存为 PNG 数据，客户端解码后绘制到监控窗口。
- 远程文件管理：支持磁盘列表、目录遍历、文件下载、删除和运行文件等命令。
- 远程控制：支持鼠标移动、点击、双击、按下、抬起，以及锁机和解锁命令。
- 通道拆分优化：客户端和服务端都拆分出 `task / screen / control` 三条链路，避免截图任务阻塞鼠标控制。

## 架构设计

### 服务端 RemoteCtrl

服务端核心在 `RemoteCtrl/EdoyunServer.*`：

- 监听地址默认是 `0.0.0.0:9527`。
- 启动时初始化 Winsock，创建重叠 I/O socket。
- 通过 `CreateIoCompletionPort` 创建完成端口。
- 使用 `AcceptEx` 异步接入客户端连接。
- 对已连接 socket 投递 `WSARecv`，接收到数据后进入协议解析。
- 业务响应通过发送队列和 `WSASend` 异步回传。

命令执行不直接堆在 IOCP 工作线程里，而是按命令类型分发到三条执行线：

- `task`：文件列表、下载、删除、测试连接等普通任务。
- `screen`：屏幕截图和画面传输。
- `control`：鼠标控制、锁机、解锁。

这样做的目的是降低截图这类重任务对鼠标控制的影响，让控制命令的响应更稳定。

### 客户端 RemoteClient

客户端核心在 `RemoteClient/ClientSocket.*` 和 MFC 对话框代码：

- 主窗口负责输入服务端 IP 和端口，发起连接测试、目录刷新、文件操作等命令。
- 监控窗口负责显示服务端回传的屏幕画面，并把本地鼠标事件转换为远程控制命令。
- 网络层按命令号分流到 `Task`、`Screen`、`Control` 三个通道，每个通道有独立队列和工作线程。
- 屏幕请求采用“上一帧显示完成后再请求下一帧”的方式，避免截图请求无限堆积。
- 鼠标移动命令支持覆盖队尾，只保留最新移动位置，减少快速移动时的控制延迟。

## 自定义协议

协议结构由 `CPacket` 封装，服务端定义在 `RemoteCtrl/Packet.h`，客户端定义在 `RemoteClient/ClientSocket.h`。

```text
| 包头 0xFEFF | 数据长度 | 命令字 | 数据区 | 校验和 |
```

字段作用：

- 包头：用于定位一个协议包的起始位置。
- 数据长度：表示命令字、数据区和校验和的总长度。
- 命令字：区分文件、截图、鼠标、锁机等业务类型。
- 数据区：存放业务参数或响应数据。
- 校验和：对数据区做简单校验，过滤异常包。

接收端不会假设一次 `recv` 就拿到完整业务包，而是在缓存上循环解析：

1. 把新收到的字节追加到接收缓存。
2. 根据包头和长度尝试解析一个完整包。
3. 如果数据不够，保留缓存等待下一次接收。
4. 如果解析成功，消费已处理字节并继续解析后续数据。

## 构建与运行

### 环境要求

- Windows 10/11
- Visual Studio 2022
- 工作负载：`使用 C++ 的桌面开发`
- 组件：MSVC v143、Windows SDK、MFC/ATL 支持

### 编译

1. 使用 Visual Studio 打开 `RemoteCtrl.sln`。
2. 选择 `Debug | x64` 或 `Release | x64`。
3. 生成整个解决方案。

### 本地演示

1. 先启动 `RemoteCtrl`，作为被控端/服务端。
2. 再启动 `RemoteClient`，作为控制端/客户端。
3. 客户端默认连接 `127.0.0.1:9527`，可以在界面里修改为被控端 IP。
4. 点击连接测试，确认通信正常后再测试目录、文件、画面和鼠标控制功能。

## 代码阅读入口

- `RemoteCtrl/RemoteCtrl.cpp`：服务端入口，启动 IOCP 服务。
- `RemoteCtrl/EdoyunServer.h`：服务端会话、IO 上下文和命令执行器定义。
- `RemoteCtrl/EdoyunServer.cpp`：IOCP 初始化、异步收发、命令分发和发送队列。
- `RemoteCtrl/Command.h`：文件、截图、鼠标、锁机等业务命令实现。
- `RemoteCtrl/Packet.h`：服务端协议包定义和解析。
- `RemoteClient/ClientSocket.h`：客户端协议包、通道状态和发送请求结构。
- `RemoteClient/ClientSocket.cpp`：客户端三通道发送、收包解析和 UI 消息回调。
- `RemoteClient/RemoteClientDlg.cpp`：客户端主窗口逻辑。
- `RemoteClient/CWatchDialog.cpp`：画面显示和鼠标事件发送。

## 项目亮点

- 使用 IOCP 完成端口模型实现 Windows 异步网络服务端，避免一连接一线程带来的线程数量膨胀。
- 用自定义 `OVERLAPPED` 上下文保存操作类型、缓冲区和会话对象，便于在完成事件返回后恢复业务上下文。
- 使用累计缓存和循环解析处理 TCP 粘包、半包，不依赖一次 `recv` 的边界。
- 将截图、控制、普通任务拆成独立链路，缓解画面传输对鼠标控制的阻塞。
- 客户端通过 MFC 界面完成命令发起、文件展示、屏幕显示和交互控制，具备完整演示闭环。

## 当前限制

- 项目依赖 Windows API、MFC 和 Winsock，不支持 Linux/macOS。
- 当前协议未加入认证、加密和权限校验，不适合部署到公网。
- 画面传输以功能演示为主，仍有压缩率、帧率和增量传输等优化空间。
